[0] QUICK INDEX
Lagless is an Nx monorepo for deterministic ECS simulation and input-synced multiplayer.
Clients run the simulation with rollback; servers only coordinate matchmaking and relay inputs.
Binary schemas and codegen keep ECS memory layouts and inputs consistent across client and server.
Circle Sumo is the reference game, with dedicated simulation, frontend, and backend apps.

Modules:
- libs/core - deterministic ECS runtime (memory layout, simulation loop, rollback, input providers)
- libs/binary - fixed-size binary serialization and schema helpers
- libs/net-wire - wire protocol structs, timing sync, input delay, tick input buffer
- libs/relay-input-provider - client-side Colyseus input provider + matchmaking helper
- libs/colyseus-rooms - server-side base rooms for matchmaking and relay input fanout
- libs/math - deterministic math ops and vectors
- libs/misc - simulation clock, snapshot history, UUIDs, interpolation helpers
- libs/animate - UI animation helpers
- libs/pixi-react - Pixi + React helpers (VFX, joystick)
- libs/react - frontend auth and API helpers
- libs/api/schemas - TypeORM entities for backend persistence
- libs/api/player - player auth/guard/service
- libs/api/game - game and matchmaking persistence services
- libs/testbed - generated ECS schema sandbox
- tools/codegen - YAML-to-TS ECS code generator (Nx generator + CLI)
- circle-sumo/circle-sumo-simulation - Circle Sumo ECS ruleset + generated schema
- circle-sumo/circle-sumo-game - Circle Sumo frontend client (React + Pixi)
- circle-sumo/circle-sumo-backend - Circle Sumo backend (Colyseus + NestJS)

Where to look for:
- main loop/tick: libs/core/src/lib/ecs-simulation.ts, circle-sumo/circle-sumo-game/src/app/game-view/runner-provider.tsx

# `@lagless/circle-sumo-backend`

## What it is
`@lagless/circle-sumo-backend` is the Colyseus + NestJS backend for Circle Sumo. It runs matchmaking, relay rooms, and persistence services for the authoritative simulation.

## Why it exists / when to use it
Use it to host Circle Sumo matches in a production-like environment. It coordinates player auth, matchmaking sessions, and relay room lifecycles.

## Public API
- Colyseus rooms: `CircleSumoMatchmakingRoom`, `CircleSumoRelayRoom`
- Nest module: `AppModule` (internal composition)
- Entry point: `circle-sumo/circle-sumo-backend/src/main.ts`

## Typical usage
`main.ts` registers the matchmaking and relay rooms:

```ts
import { CircleSumoMatchmakingRoom } from './colyseus/matchmaking';
import { CircleSumoRelayRoom } from './colyseus/relay';

gameServer.define('matchmaking', CircleSumoMatchmakingRoom);
gameServer.define('relay', CircleSumoRelayRoom);
```

## Key concepts & data flow
- Matchmaking rooms authenticate players and create game sessions via `GameService` and `PlayerService`.
- Relay rooms extend `RelayColyseusRoom` and feed inputs to the Circle Sumo simulation using `CircleSumoInputRegistry`.
- Nest services handle persistence (`@lagless/schemas`) while Colyseus handles WebSocket traffic.

## Configuration and environment assumptions
- `PORT` controls the server port.
- `JWT_SECRET` is required for matchmaking auth.
- `DB_CONNECTION_STRING` is required for TypeORM.
- Matchmaking and relay frame length should align with frontend `ECSConfig` (60 FPS in Circle Sumo).

## Pitfalls / common mistakes
- Blocking I/O in Colyseus room callbacks (causes tick drift).
- Mismatched frame length or input registry between backend and frontend.
- Forgetting to register schemas in TypeORM modules.

## Related modules
- `libs/colyseus-rooms`, `libs/game`, `libs/player`, `libs/schemas`
- `circle-sumo/circle-sumo-simulation` for the ECS ruleset
- `libs/relay-input-provider` for the client counterpart

## What it is

import { toFloat32 } from '@lagless/binary';

# `@lagless/core`

## What it is
`@lagless/core` is the deterministic ECS runtime for Lagless. It owns the memory model, simulation loop, rollback/resimulation flow, and DI wiring that every game simulation depends on.

## Why it exists / when to use it
Use it when you are building a simulation, input provider, or tool that must run deterministically and support rollback. It is the authoritative source of world state; rendering and networking sit on top of it.

## Public API
- `ECSConfig`: configuration for ticks, snapshot cadence, and rollback window
- `ECSSimulation`: deterministic loop with rollback/resimulation
- `ECSRunner`: DI bootstrapper for systems, signals, and input provider
- `AbstractInputProvider`, `LocalInputProvider`, `ReplayInputProvider`, `InputRegistry`
- `RPC`, `RPCHistory`: input payload and history tracking
- `Signal`, `ISignalConstructor`: predicted/verified/cancelled signal flow
- `Prefab`, `EntitiesManager`, `PlayerResources`, `PRNG`: ECS memory helpers

## Typical usage
Circle Sumo wires a generated runner with relay inputs and ticks it each frame:

```ts
import { CircleSumoRunner, CircleSumoSystems, CircleSumoSignals } from '@lagless/circle-sumo-simulation';
import { RelayInputProvider } from '@lagless/relay-input-provider';

const runner = new CircleSumoRunner(inputProvider.ecsConfig, inputProvider, CircleSumoSystems, CircleSumoSignals);
runner.start();
useTick((ticker) => runner.update(ticker.deltaMS));
```

## Key concepts & data flow
- `ECSRunner` builds a DI container, registers component managers, and instantiates systems and signals.
- `ECSSimulation` advances ticks using a fixed frame length and keeps snapshots for rollback.
- `AbstractInputProvider` supplies ordered RPC inputs and can request rollback when authoritative inputs arrive late.
- Signals emit `Predicted`, `Verified`, and `Cancelled` events tied to rollback windows.

## Configuration and environment assumptions
- `ECSConfig` values (fps, snapshotRate, snapshotHistorySize, input delays) are fixed for a session.
- Deterministic math must be initialized (`MathOps.init()` from `@lagless/math`) before ticking.
- Input providers must supply inputs sorted by tick and never mutate ECS memory directly.

## Pitfalls / common mistakes
- Introducing non-determinism (`Math.random`, Date/time APIs) inside systems.
- Keeping snapshot history smaller than the maximum rollback window.
- Mutating component arrays outside system update methods.

## Related modules
- `libs/math` and `libs/misc` for deterministic math and snapshot helpers.
- `tools/codegen` for generated ECS components and runners.
- `libs/relay-input-provider` and `libs/net-wire` for networked inputs.

# `@lagless/net-wire`

## What it is
`@lagless/net-wire` defines the shared binary wire protocol for Lagless input-only networking. It includes packet schemas, clock sync helpers, and input delay logic.

## Why it exists / when to use it
Use it anywhere you need to encode or decode Lagless relay packets or compute client-side input delay. It keeps the client and server in sync on the exact byte layout and timing rules.

## Public API
- `WireVersion`, `MsgType`, `TickInputKind`, `RELAY_BYTES_CHANNEL`
- Binary schemas: `HeaderStruct`, `ServerHelloStruct`, `TickInputStruct`, `TickInputFanoutStruct`, `CancelInputStruct`, `PingStruct`, `PongStruct`, `PlayerFinishedGameStruct`
- `ClockSync`: RTT/jitter tracking and clock offset estimation
- `InputDelayController`: converts RTT/jitter into target input delay
- `TickInputBuffer`: stores recent inputs for late joiners
- `ColyseusRelayRoomOptions`: relay room connection options

## Typical usage
Circle Sumo does not import `@lagless/net-wire` directly; the relay server and input provider do. This is the pattern used inside those layers:

```ts
import { HeaderStruct, MsgType, TickInputStruct, WireVersion } from '@lagless/net-wire';

const buffer = new ArrayBuffer(HeaderStruct.byteLength + TickInputStruct.byteLength);
const header = HeaderStruct.pack(buffer);
header.struct.version = WireVersion.V1;
header.struct.type = MsgType.TickInput;
```

## Key concepts & data flow
- Every packet begins with `HeaderStruct` and a `MsgType` identifier.
- `ClockSync` tracks RTT and jitter via Ping/Pong to align client tick timing.
- `InputDelayController` clamps delay inside configured min/max bounds.

## Configuration and environment assumptions
- All schemas are fixed-size and little-endian via `@lagless/binary`.
- Client and server must share the same `WireVersion` and `MsgType` list.

## Pitfalls / common mistakes
- Changing schema ordering without bumping `WireVersion`.
- Sending ad-hoc JSON or string payloads on the relay bytes channel.
- Using clock sync values before receiving enough Pong samples.

## Related modules
- `libs/relay-input-provider` uses these schemas on the client.
- `libs/colyseus-rooms` uses these schemas on the server.
- `circle-sumo/circle-sumo-backend` and `circle-sumo/circle-sumo-game` consume those layers.

- **Determinism and timing**:
  - ECSConfig defines fixed fps, frameLength, rollback window, snapshot cadence, and input delay bounds.
  - MathOps wraps deterministic math (requires MathOps.init); Circle Sumo calls it in AssetsLoader.
  - PRNG is seeded via server hello (seed0/seed1) and uses deterministic xoshiro state in ECS memory.
  - SimulationClock + PhaseNudger apply drift correction based on clock sync.
- **State representation and serialization**:
  - Mem owns a single ArrayBuffer for all ECS state and can export/apply snapshots (full memory copy).
  - SnapshotHistory stores snapshots by tick and supports rollback truncation.
  - RPCHistory can export/import a binary input history for replay files.
  - InputBinarySchema packs/unpacks input payloads into fixed-size binary buffers.
- **Signals/events**:
  - SignalsRegistry calls Signal lifecycle hooks each tick and before rollback.
  - Circle Sumo signals (GameOver, PlayerFinishedGame, HighImpact) are consumed by the frontend for UI and analytics.
- **Networking and relay**:
  - net-wire defines the binary protocol on RELAY_BYTES_CHANNEL (99).
  - RelayInputProvider handles Ping/Pong clock sync and adaptive input delay.
  - RelayColyseusRoom is input-only; it never runs ECS simulation.
- **Circle Sumo wiring**:
  - use-start-match.ts uses Matchmaking + RelayInputProvider.connect and stores the provider for the game view.
  - RunnerProvider builds CircleSumoRunner with CircleSumoSystems and CircleSumoSignals and hooks PlayerFinishedGameSignal to send to the server.
  - DirectionArrowView drains local pointer inputs and emits LookAt/Move RPCs.
- **UNKNOWN**:
  - No existing majority-vote late-join snapshot flow; only TickInputBuffer-based input replay is present (see libs/colyseus-rooms/src/lib/relay-colyseus-room.ts and libs/relay-input-provider/src/lib/relay-input-provider.ts).

  public update(dt: number) {
    this.clock.update(dt);

    const targetTick = Math.floor(this.clock.accumulatedTime / this._frameLength);

    this.checkAndRollback(this.mem.tickManager.tick);
    this.simulationTicks(this.mem.tickManager.tick, targetTick);

    this._inputProvider.update();

    const simTick = this.mem.tickManager.tick;
    const tickTime = simTick * this._frameLength;
    const leftover = this.clock.accumulatedTime - tickTime;

    this._interpolationFactor = MathOps.clamp01(leftover / this._frameLength);
  }

  private checkAndRollback(currentTick: number) {
    const rollbackTick = this._inputProvider.getInvalidateRollbackTick();

    if (rollbackTick === undefined || rollbackTick > currentTick) return;

    this.rollback(rollbackTick);
  }

  private simulationTicks(currentTick: number, toTick: number): void {
    if (toTick - currentTick > 1) {
      console.warn(`Simulation ticks: ${currentTick} -> ${toTick} (simulate ${toTick - currentTick})`);
    }

    while (currentTick < toTick) {
      this.mem.tickManager.setTick(++currentTick);
      this.simulate(currentTick);
      this._signalsRegistry.onTick(currentTick);
      this.storeSnapshotIfNeeded(currentTick);
      for (const handler of this._onTickHandlers) handler(currentTick);
    }
  }

  ...
  private storeSnapshotIfNeeded(tick: number): void {
    if (this._snapshotRate === 0) return;

    if (tick % this._snapshotRate === 0) {
      this.saveSnapshot(tick);
    }

    if (tick % 200 === 0) {
      console.log(`Mem Hash at tick ${tick}: ${this.mem.getHash()}`);
    }
  }

export class InputBinarySchema {
  public static packBatch(
    registry: { get(id: number): { id: number; fields: ReadonlyArray<InputFieldDefinition>; byteLength: number } },
    data: ReadonlyArray<{ inputId: number; ordinal: number; values: { [key: string]: number | ArrayLike<number> } }>
  ): ArrayBuffer {
    const totalByteLength = data.reduce((acc, { inputId }) => {
      const struct = registry.get(inputId);
      return (
        acc +
        fieldTypeSizeBytes[FieldType.Uint8] + // input id
        fieldTypeSizeBytes[FieldType.Uint32] + // ordinal
        struct.byteLength // size of payload for this struct
      );
    }, 0);

    const buffer = new ArrayBuffer(totalByteLength);
    const dataView = new DataView(buffer);
    let offset = 0;

    for (const { inputId, ordinal, values } of data) {
      const struct = registry.get(inputId);

      // write input id
      binaryWrite(dataView, offset, FieldType.Uint8, struct.id);
      offset += fieldTypeSizeBytes[FieldType.Uint8];

      // write ordinal
      binaryWrite(dataView, offset, FieldType.Uint32, ordinal);
      offset += fieldTypeSizeBytes[FieldType.Uint32];

      // write struct fields
      for (const field of struct.fields) {
        if (field.isArray) {
          const rawValue = values[field.name] as ArrayLike<number>;
          if (!rawValue || rawValue.length === undefined) throw new Error(`Expected array for field ${field.name}`);

          const arrayLength = rawValue.length;
          const expectedLength = field.arrayLength;

          if (expectedLength === undefined) {
            throw new Error(`Cannot pack variable-length array without declared arrayLength for field ${field.name}`);
          }
          if (arrayLength > expectedLength) {
            throw new Error(
              `Array length for field ${field.name} exceeds expected length (${arrayLength} > ${expectedLength})`
            );
          }

          // Ensure we have room for the *declared* payload (writer may underfill; zero-init covers the rest)
          ensureCapacity(dataView, offset, field.byteLength, `InputBinarySchema.packBatch field ${field.name}`);

          // Write provided elements
          for (let j = 0; j < arrayLength; j++) {
            const rawElement = rawValue[j];
            if (rawElement === undefined || rawElement === null || !Number.isFinite(rawElement) || isNaN(rawElement))
              throw new Error(`Unexpected value "${rawElement}" at index ${j} for field ${field.name}`);
            const element = throwIfOutOfBounds(`${field.name}[${j}]`, field.type, rawElement);
            binaryWrite(dataView, offset, field.type, element);
            offset += fieldTypeSizeBytes[field.type];
          }

          // Skip remaining bytes for padding (buffer is zero-initialized)
          const elementSize = fieldTypeSizeBytes[field.type];
          const padCount = expectedLength - arrayLength;
          offset += padCount * elementSize;
        } else {
          const rawValue = values[field.name];
          if (typeof rawValue !== 'number') throw new Error(`Expected number for field ${field.name}`);
          if (!Number.isFinite(rawValue) || isNaN(rawValue))
            throw new Error(`Unexpected value "${rawValue}" for field ${field.name}`);
          const element = throwIfOutOfBounds(field.name, field.type, rawValue);
          ensureCapacity(
            dataView,
            offset,
            fieldTypeSizeBytes[field.type],
            `InputBinarySchema.packBatch field ${field.name}`
          );
          binaryWrite(dataView, offset, field.type, element);
          offset += fieldTypeSizeBytes[field.type];
        }
      }
    }

    return buffer;
  }

    registry: { get(id: number): { id: number; fields: ReadonlyArray<InputFieldDefinition>; byteLength: number } },

  private handleTickInputFanout(pipeline: BinarySchemaUnpackPipeline): void {
    pipeline.unpack(TickInputFanoutStruct);
    const inputsBuffer = pipeline.sliceRemaining();

    if (inputsBuffer.byteLength === 0) return;

    const unpackedBuffers = unpackBatchBuffers(inputsBuffer);
    const receivedRpcs: RPC[] = [];
    let minReceivedTick = Number.MAX_SAFE_INTEGER;

    for (const buf of unpackedBuffers) {
      const bufPipeline = new BinarySchemaUnpackPipeline(buf);
      const tickInput = bufPipeline.unpack(TickInputStruct);

      // Skip own client inputs (we already have them in local history)
      if (tickInput.playerSlot === this.playerSlot && tickInput.kind === TickInputKind.Client) {
        continue;
      }

      // Track minimum tick for rollback calculation
      if (tickInput.tick < minReceivedTick) {
        minReceivedTick = tickInput.tick;
      }

      // Unpack input payload
      const payloadBuffer = bufPipeline.sliceRemaining();
      const rawInputs = InputBinarySchema.unpackBatch(this._inputRegistry, payloadBuffer);

      for (const rawInput of rawInputs) {
        const rpc = new RPC(
          rawInput.inputId,
          {
            tick: tickInput.tick,
            playerSlot: tickInput.playerSlot,
            seq: tickInput.seq,
            ordinal: rawInput.ordinal,
          },
          rawInput.values
        );
        receivedRpcs.push(rpc);
      }
    }

    // Add to history and trigger rollback if needed
    if (receivedRpcs.length > 0) {
      this._rpcHistory.addBatch(receivedRpcs);

      if (minReceivedTick < Number.MAX_SAFE_INTEGER) {
        this.requestRollback(minReceivedTick);
      }
    }
  }

  private sendFrameInputs(): void {

    const firstRpc = this._frameRPCBuffer[0];

    // Pack all inputs from this frame
    const packedInputs = InputBinarySchema.packBatch(
      this._inputRegistry,
      this._frameRPCBuffer.map(rpc => ({

        ordinal: rpc.meta.ordinal,
        values: rpc.data,

      playerSlot: this.playerSlot,
      kind: TickInputKind.Client,
      seq: firstRpc.meta.seq,
    });
    pipeline.appendBuffer(packedInputs);

    this._room.send(RELAY_BYTES_CHANNEL, pipeline.toUint8Array());

[4] MULTIPLAYER V2 DESIGN (SPEC + API)
A) Concepts & Goals
- Input-sync lockstep: gameplay traffic is input-only; the only non-input payloads are late-join snapshots.
- Server is coordinator only: matchmaking, slot assignment, input relay, and late-join snapshot arbitration (no ECS simulation).
- Latency strategy: delay-based local input (ECSConfig.initialInputDelayTick) plus rollback on late authoritative inputs, matching the existing RelayInputProvider/ECSSimulation model.
- Fairness and stability: fixed tick cadence, ordered RPCs (tick + playerSlot + ordinal), late input cancellation on the server, and optional state hash checks for desync detection.

B) Roles & Components
- Client runtime
  - MultiplayerClient (new wrapper) manages matchmaking/session lifecycle, exposes connection state, and owns a MultiplayerInputProvider (AbstractInputProvider implementation).
  - MultiplayerInputProvider handles binary protocol, ping/pong timing, adaptive input delay, rollback requests, and snapshot request/response.
  - Optional spectator mode: read-only client that receives snapshot + input fanout but never sends TickInput.
- Server relay/matchmaking
  - Matchmaking room (Colyseus) supports quick match, create room, join by code, and filters.
  - Relay room (Colyseus) relays TickInput messages, maintains TickInputBuffer, and executes late-join snapshot vote aggregation.

C) Data Model
- InputFrame (wire-level unit of simulation)
  - tick: number (simulation tick index)
  - playerSlot: number
  - seq: number (monotonic per-client sequence)
  - ordinal: number (order within a tick)
  - kind: 'client' | 'server'
  - payload: Uint8Array (InputBinarySchema pack of one or more inputs)
- Server room state (no simulation state stored)
  - gameId, frameLength, maxPlayers, roomStartTime
  - players: sessionId -> { playerSlot, isConnected, joinedAtTick, lastPingMs, version, flags }
  - inputBuffer: TickInputBuffer (recent inputs for late joiners)
  - rpcHistory: RPCHistory (optional for replay persistence)
  - snapshotVotes: Map<requestId, Map<hash, { tick, count, senders[] }>>
- Late-join snapshot model (client-provided)
  - SnapshotCandidate:
    - requestId: string
    - tick: number (snapshot tick)
    - hash32: number (hash of snapshot bytes or Mem.getHash)
    - sizeBytes: number
    - senderSlot: number
    - bytes: Uint8Array (full Mem snapshot or compressed chunk)
  - LateJoinBundle:
    - snapshotTick: number
    - snapshotHash: number
    - snapshotBytes: Uint8Array
    - serverTick: number
    - inputsFromTick: Uint8Array[] (TickInput buffers from snapshotTick+1..serverTick)

D) Protocol
- Client -> server (binary on RELAY_BYTES_CHANNEL)
  - TickInput (existing)
  - Ping (existing)
  - PlayerFinishedGame (existing)
  - SnapshotResponse (new): requestId, snapshotTick, hash32, chunkIndex/chunkCount, snapshotBytes
  - ClientReady (new, optional): client version, schema hash, requested role (player/spectator)
- Server -> client (binary on RELAY_BYTES_CHANNEL)
  - ServerHelloV2 (new): seed0/seed1, playerSlot, serverTick, roomConfig, wireVersion
  - TickInputFanout (existing)
  - CancelInput (existing)
  - Pong (existing)
  - SnapshotRequest (new): requestId, targetTick or targetWindow, maxBytes, preferredChunkSize
  - LateJoinBundle (new): snapshotTick, snapshotHash, snapshotBytes, inputsFromTick
  - RoomClosing (new): reason, finalTick
- Heartbeats and disconnects
  - Ping/Pong already drive ClockSync and InputDelayController.
  - Server maintains lastPingMs and disconnect timeout; mark player as disconnected, allow rejoin within a grace window (same auth token + slot).
- Backpressure and rate limiting
  - Server drops TickInput that exceed rate limits (per-client max inputs per tick) and logs warnings.
  - SnapshotResponse is chunked and capped by maxBytes per message; late join request rejects overly large snapshots.
- Idempotency and ordering
  - TickInput ordering uses tick + playerSlot + ordinal (RPCHistory sorts by slot+ordinal).
  - SnapshotRequest/Response use requestId to dedupe and ignore stale chunks.
  - LateJoinBundle is only applied once per join attempt; subsequent bundles are ignored unless requestId changes.

E) Late-Join via Majority Vote (CORE)
- Snapshot production (clients)
  - On SnapshotRequest, each client chooses a snapshot tick that it already has stored (e.g., latest tick % snapshotRate == 0) and within server input buffer retention.
  - Client serializes state using ECSSimulation.mem.exportSnapshot() and computes hash32 (Mem.getHash or fast hash of snapshot bytes).
  - Client sends SnapshotResponse with (tick, hash, bytes) in chunks if needed.
- Majority selection (server)
  - Server groups SnapshotResponse messages by (snapshotTick, hash32).
  - Majority threshold: at least floor(connectedPlayers/2) + 1, with a configurable minimum (e.g., minVotes = 2).
  - Tie-breakers: prefer the candidate with highest tick (more recent), then earliest complete arrival, then lowest hash (deterministic fallback).
  - If no candidate reaches threshold within timeout, server retries once with a new request or rejects the late join with a clear error.
- Malicious/buggy client handling (lightweight)
  - Ignore snapshots with size mismatches, invalid tick window, or hash/byte length inconsistencies.
  - Track offenders and exclude them from future snapshot requests for this room.
  - No heavy crypto required; rely on majority and simple validation.
- Late joiner catch-up
  - Late joiner receives LateJoinBundle with snapshot at tick T and input buffers from T+1..serverTick.
  - Client applies snapshot via mem.applySnapshot(), then replays buffered inputs into RPCHistory and advances simulation to serverTick.
  - After catch-up, client enters normal TickInputFanout flow and begins sending its own inputs at serverTick + inputDelay.

F) Matchmaking (CONFIGURABLE BUT SIMPLE)
- Minimal modes
  1) Create room: generates a private code, returns a seat reservation.
  2) Join by code: validates code and returns a seat reservation.
  3) Quick match: uses MatchmakingService queue with optional filters.
- Configuration knobs
  - maxPlayers, region, ruleset, version, allowLateJoin, private/public, roomCodeLength, roomTTL, preferredInputDelay.
  - For quick match: reuse MatchmakingConfig (mmr/ping windows, startDelayByHumans, virtualCapacity).
- End of game flow
  - Clients emit PlayerFinishedGame (existing) with verifiedTick.
  - Relay room waits for all connected players to finish or disconnect timeout, then triggers onBeforeDispose and broadcasts RoomClosing.
  - Persist results in GameService (existing Circle Sumo pattern) and close room.

G) Integration Points with lagless
- Input capture: use AbstractInputProvider.drainInputs (circle-sumo direction-arrow-view.tsx is the pattern).
- Input buffer: AbstractInputProvider stores local RPCs in RPCHistory; RelayInputProvider adds remote RPCs to the same history.
- Applying remote inputs: RelayInputProvider.handleTickInputFanout unpacks InputBinarySchema and adds to RPCHistory, then requests rollback.
- Tick advancement gating: ECSSimulation.update calls inputProvider.getInvalidateRollbackTick to trigger rollback and resim.
- Optional checksum verification: use Mem.getHash per N ticks and exchange via a lightweight MsgType.StateHash (new) for desync detection.
- Circle Sumo adoption steps (high level)
  - Update use-start-match.ts to use MultiplayerClient (create/join/quick match) and return MultiplayerInputProvider.
  - Update RunnerProvider to handle late-join bootstrap (apply snapshot + replay inputs before starting ticks).
  - Update backend relay to use the V2 relay room and enable allowLateJoin.
