Я сейчас работаю над ECS-фреймворком для своих игр.
Это детерминированный фреймворк на TypeScript, с simulate/rollback подходом.
Всё состояние симуляции хранится в ArrayBuffer.
Основные идеи фреймворка:
- Детерминированность: при одинаковых входных данных, симуляция всегда будет одинаковой
- State-oriented: всё состояние симуляции хранится в ArrayBuffer, что позволяет легко сохранять и загружать состояние. А так же отправлять другим клиентам.
- Производительность
- Отсутствие View системы: рендеринг и логика симуляции полностью разделены. Фреймворк не знает ничего о том, как рендерить объекты, он просто предоставляет данные для рендеринга.
- Multiplayer-Ready: фреймворк делается с прицелом на мультиплеерные игры.
Основной схемой мультиплеера должен являться Relay Server (не реализовано), который является источником времени симуляции, принимает инпут игроков, проверяет чтоб инпут пришел вовремя,
если нет, то не отправляет в ответ игроку CancelInput, клиент игрока делает rollback
и удаляет инпут и накатывает симуляцию заново. Симуляция на сервере - не запускается.
Глобально, Relay Multiplayer должен состоять из RelayInputProvider (наследует AbstractInputProvider) и непосрественно сервера.
Очень важные фичи для Relay Multiplayer:
- Поддержка Late-Join: игроки должны иметь возможность присоединиться к игре в любой момент, даже если она уже началась.
Тогда сервер запрашивает у клиентов их state, при возможности проводит majority vote и отправляет state игроку, который присоединился. Если majority vote провести не получается, то сервер может отправить state от одного из клиентов, который присоединился раньше всех.
- Поддержка Reconnect: если игрок потерял соединение, он должен иметь возможность переподключиться и продолжить игру с того места, на котором он остановился, тоже по сути late-join, но возможно это только если в матче остались другие игроки.
- Server-Side Events: это Inputs, которые может слать только server
- Сервер дожен быть легко расширяемым, скорее всего должен быть сделан абстрактным классом или как то еще (подумай над этим хорошо, когда дойдём до этого)
- Завершение игры - нужно поддерживать множество условий завершения игры:
  - все игроки отключились
  - все игроки отправили Input что завершили игру (как правило с результатом)
  - тут важно иметь возможность управлять поведением из класса который расширяет абстрактный класс сервера (или какой там будет), то есть часто нужно будет записывать игровую сессию и результат игры в базу, но всегда будут разные схемы данных для каждой из игр.
